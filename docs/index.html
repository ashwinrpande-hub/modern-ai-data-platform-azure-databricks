Architecture â€” acme Sales Lakehouse on Azure Databricks

Source landscape â†’ Bronze (templatized, see REPLICATION_PATTERNS.md)

SAP S/4HANA (EU) & Oracle EBS â†’ GoldenGate â†’ ADLS â†’ Auto Loader/AUTO CDC SQL Server (QAD host) â†’ Lakeflow Connect SQL Server (CDC) â€” replaces Fabric Mirroring Salesforce â†’ file/API landing â†’ Auto Loader OT (PLCs/SCADA) â†’ Litmus Edge UNS â†’ Event Hubs (Kafka) or ADLS parquet â†’ Structured Streaming

Medallion â€” Data Vault 2.0 methodology (see DATA_VAULT_PIT_BRIDGE.md)

The lakehouse applies DV2.0 methodology without materializing hub/link/sat objects: - Business keys â†’ hash keys: hk = sha2(source_system|pk, 256) in Silver plays the DV hub-key role â€” deterministic, source-agnostic, computed not looked-up. - Satellites â†’ insert-only versions: every Silver change is a new row with effective_ts; full history, no UPDATE. record_hash = DV hashdiff (no-op change detection). - Links â†’ virtual: relationships live as hash-key FKs inside 3NF entities, not link tables. - Query-assistance tables â†’ materialized in Gold: pit_customer (point-in-time) and bridge_order_fulfillment (orderâ†’shipmentâ†’invoice) â€” the DV2.0 answer to "history is cheap to store, expensive to query."

Layers

  • Bronze (acme_bronze) â€” schema-on-write mirror of source 3NF structures; envelope columns (_source_system,_ingest_ts,_batch_id,_op_type,_seq,_raw); raw payload preserved in _raw (schema-on-read escape hatch). DLT expectations quarantine bad rows.
  • Silver (acme_silver) â€” 3NF, insert-only, DV-aligned as above; "current" exposed as a view (row_number over hk), never via UPDATE. FX-normalized to USD. DAMA DQ gates.
  • Gold (acme_gold) â€” star schema (dims + facts joined on hash keys), DV2.0 PIT + bridge query-assist tables, pre-aggregations for AI/BI + Genie, ML feature store (PIT-correct via pit_customer), vector indexes for semantic search/RAG.
  • Products (acme_products) â€” secure-view output ports, Delta Sharing, Marketplace listings.

Databricks features used (and why)

Lakeflow Declarative Pipelines (DLT) â€” expectations, AUTO CDC, serverless; Lakeflow Connect â€” managed SQL Server CDC; Lakeflow Jobs â€” orchestration; Unity Catalog â€” RBAC, row filters/masks, lineage, tags; Liquid Clustering â€” fact pruning; Predictive Optimization â€” auto OPTIMIZE/VACUUM; Genie â€” NL Q&A on agg tables; Agent Bricks / Mosaic AI â€” DQ RCA + healer agents, RAG serving; Online/Lakebase â€” low-latency features; Databricks Apps â€” marketplace portal + agent hosting; Delta Sharing/Marketplace â€” product distribution.

Data Vault 2.0 â€” PIT & Bridge use cases in Gold

How DV2.0 concepts map onto this lakehouse (methodology adopted; hub/link/sat objects not materialized â€” reasoning in JUSTIFICATION_NOTES #5):

DV2.0 concept Here
Hub business key (hashed) Silver hk = sha2(source_system\|pk, 256)
Satellite (history, hashdiff) Silver insert-only versions: effective_ts, record_hash
Link Virtual â€” hash-key FKs inside 3NF entities
PIT table acme_gold.sales.pit_customer
Bridge table acme_gold.sales.bridge_order_fulfillment
Information mart Gold star (dims/facts) + aggs

Use case 1 â€” pit_customer (point-in-time)

Grain: one row per (snapshot_date, customer_hk); as_of_ts points at the Silver customer version that was current on that date. Daily spine from first customer version to today.

Solves: "history is cheap to store, expensive to query." Without PIT, every as-of question is a range join over all versions â€” easy to write wrong (label leakage) and slow at scale.

  1. Leakage-free ML training sets â€” ml/feature_store.py joins labels to features via pit_customer at label_date; the model never sees attributes newer than the label.
  2. Reproducible month-end reporting â€” re-running a January report in July returns January's customer attributes (name/country/region as they were), not today's.
  3. Audit / region policy evidence â€” prove which region row-filter applied to a customer on a given date (source_system was X on date D).
-- as-of join: 2 equality joins, no BETWEEN scan
SELECT l.label_date, c.customer_name, c.country
FROM   labels l
JOIN   acme_gold.sales.pit_customer p
       ON p.customer_hk = l.customer_hk AND p.snapshot_date = l.label_date
JOIN   acme_silver.sales.customer c
       ON c.hk = p.customer_hk AND c.effective_ts = p.as_of_ts;

Use case 2 â€” bridge_order_fulfillment (order â†’ shipment â†’ invoice)

Grain: one row per current order, pre-joined to its shipment and invoice hash keys (NULL until each lifecycle stage exists). Higher-level "table of keys + measures of interest" so consumers make ONE join instead of walking the lifecycle.

  1. Order-to-cash cycle time â€” datediff(invoice_date, order_date) per region/source, no 3-table join for analysts or Genie.
  2. Backlog dashboards â€” is_shipped = false / is_invoiced = false flags give unshipped and shipped-not-billed backlog with a WHERE clause.
  3. Genie/NL-friendly â€” natural-language questions ("orders not yet invoiced in EMEA") resolve against one table; no join-path hallucination.
SELECT source_system,
       avg(datediff(invoice_date, order_date)) AS order_to_cash_days,
       count_if(NOT is_shipped)                AS backlog_orders
FROM   acme_gold.sales.bridge_order_fulfillment
GROUP  BY source_system;

Load pattern & scope notes

  • Both are batch materialized views in pipelines/gold_dlt.py, recomputed per pipeline run â€” structural logic stays code (same precedent as furnace_heat_5min); entity/column semantics stay in cfg.layer_mappings.
  • Lifecycle sources seeded SAP-only in the pilot (LIPS â†’ shipment, VBRP â†’ invoice); JDE/QAD onboarding = mapping rows in config/seed_layer_mappings.sql, no code change.
  • Invoice currency deliberately omitted (lives on VBRK header, not VBRP item) â€” add a VBRK mapping before using invoice amounts for finance; flagged so we don't misstate SAP schema.
  • PIT spine is daily (ML need). At scale: restrict spine to business-relevant dates or switch to an incremental snapshot job appending only the new day.
  • Retention: PIT/bridge are derived â€” safe to CREATE OR REPLACE; truth stays in Silver history.

Decision log â€” each with two rejected alternatives

  1. Bronze landing via GoldenGateâ†’ADLSâ†’Auto Loader (Oracle/EBS) Why: reuses existing OGG licenses/skills; ADLS landing decouples capture from compute; avro_op carries op_type+CSN for exact replay. Alt A: Lakeflow Connect Oracle â€” cleanest long-term; adopt when GA in tenant (YAML pattern swap only). Alt B: OGGâ†’Kafkaâ†’streaming â€” lower latency, chosen only if multiple consumers need the CDC stream; otherwise extra infra.

  2. SQL Server via Lakeflow Connect, replacing Fabric Mirroring Why: single governance plane (UC), managed CDC, no OneLake hop, serverless processing. Alt A: keep Mirroring + external location/shortcut â€” cheapest, but split governance, two copies of truth. Alt B: Debezium+Event Hubs â€” max control/latency, highest run cost & on-call burden.

  3. OT via Litmus â†’ Event Hubs (default) with ADLS-parquet fallback Why: Litmus already normalizes to UNS at the edge; Event Hubs Kafka API gives sub-minute freshness to Bronze; parquet path stays for high-volume historians. Alt A: Azure IoT Operations â€” duplicates Litmus's job. Alt B: direct historian JDBC pulls â€” batch-only, loses tags/UNS.

  4. Insert-only Silver with SHA-256 hash keys Why: deterministic keys across 3 ERPs (system|pk), full history for free, audit-trivial, merge-free = fast streaming writes; "current" is a cheap window view. Alt A: SCD2 with MERGE â€” familiar, but update-heavy, slower streams, key drift risk. Alt B: identity surrogate keys â€” compact joins but non-deterministic across reloads/environments.

  5. 3NF Silver, star Gold joined on hash keys â€” DV2.0 methodology, not DV2.0 objects (Revised 2026-07-11: requirement now names Data Vault 2.0 explicitly with PIT/bridge in Gold.) Why: we adopt DV2.0's methodology â€” hashed business keys (hub keys), insert-only history with hashdiff (satellites), and materialized PIT + bridge query-assist tables in Gold â€” while keeping Silver 3NF instead of physical hub/link/sat objects: 3NF preserves source semantics, halves the object count for one domain, and Gold shapes assemble by hash key without lookups. Alt A: full raw vault (hubs/links/sats) â€” maximum auditability and multi-domain integration pattern; 3â€“4x the tables/pipelines for a single-domain pilot, defensible at enterprise scale-out. Alt B: One Big Table / plain Kimball star straight from Bronze â€” fastest to first dashboard, but loses deterministic keys, history-by-default, and the PIT/bridge answers below.

  6. Config tables for sourceâ†’target mapping at each layer Why: changes = versioned inserts (lineage built-in); agents and humans share one contract; no code edits to onboard sources. Alt A: dbt-style code-as-mapping â€” great for analysts, weaker for non-SQL OT pipelines. Alt B: hardcoded notebooks â€” fastest day 1, unscalable day 30.

  7. DQ as DAMA 6 dimensions with quarantine + thresholds that fail jobs Why: business-recognizable scoring, row-level rejects with reasons, breach triggers RCA agent. Alt A: DLT expectations only â€” great inline, but no cross-table/dimension scoring. Alt B: Great Expectations â€” rich but another runtime; revisit at multi-domain scale.

  8. Region access via UC row filters keyed on source_system Why: EUROPEâ†’SAP, AMERICASâ†’JDE, EMEAâ†’QAD maps 1:1 to source_system column; one function, applied to every Gold table. Alt A: separate regional catalogs â€” strong isolation, triples objects. Alt B: BI-tool security â€” bypassable via SQL, audit gap.

  9. Marketplace = Delta Sharing/Databricks Marketplace + thin custom portal Why: native listing handles entitlements; portal adds branded discovery + AI search without owning access control. Alt A: portal-only with direct grants â€” re-implements entitlements. Alt B: third-party catalog (Collibra/Atlan) â€” strong governance suite, heavy for a pilot.

  10. PIT as a materialized daily-spine table (pit_customer), not query-time as-of joins Why: equality joins (snapshot_date, customer_hk) replace BETWEEN range scans; one blessed as-of path kills ML label leakage; reproducible month-end reports; derived â€” safe to rebuild. Alt A: query-time AS OF / range joins on effective_ts â€” zero storage, but every consumer rewrites (and mis-writes) temporal logic; slow at scale. Alt B: SCD2 dim with valid_from/valid_to â€” familiar to BI, but needs UPDATE (breaks insert-only Silver) and still range-joins at query time.

  11. Bridge as pre-joined lifecycle keys (bridge_order_fulfillment), grain = current order Why: orderâ†’shipmentâ†’invoice in ONE join for O2C cycle time, backlog flags, and Genie (no join-path hallucination); NULLs mark incomplete lifecycle stages honestly. Alt A: multi-hop joins in the semantic/BI layer â€” no extra storage, but every tool re-encodes join paths and pays 3-table joins per query. Alt B: denormalize lifecycle into a wide fact_sales_orders â€” fastest reads, but grain explosion (order vs shipment-line vs invoice-line) and full reload whenever a late stage arrives.

Reusable Replication Patterns â†’ Bronze

Goal: any technical person lands a new source in Bronze by (1) adding one row to config/replication_sources.yaml, (2) running scripts/deploy.py --register-source <name>. No new code. Three patterns cover the estate.
